# Skill: workspace-hygiene

> The orchestrator invokes this **before trusting any local `.harness` file**. It prevents a branch-backed or
> in-place session from reusing a previous run's `.harness/plan.json`, `.harness/work-plan.md`, or
> `.harness/dispatch.json` / `.harness/units/*.json` as if it belonged to a new scenario.

## When to invoke
- At the very start of every orchestrator run, before project-zero/bootstrap decisions.
- Before `plan-to-issues`, before implementation dispatch, and whenever a refresh/retry session starts.
- Any time the agent sees an existing `.harness/plan.json` and is about to conclude "a plan already exists."

## Inputs
- Target repo path.
- `github.org/repo` and `defaultBranch` if already known; otherwise discover from `gh repo view` / `git remote`.
- Current run intent identifier if known: source intake title, source commit, tracking issue, or run timestamp.

## Procedure
1. **Fetch and compare default branch state.**
   ```bash
   git fetch --prune origin
   git status --short --branch
   git rev-parse HEAD
   git rev-parse origin/<defaultBranch>
   ```
   If the local default branch is behind or diverged from `origin/<defaultBranch>`, do **not** plan or dispatch
   until reconciled.
2. **Classify dirty worktree state.**
   - Tracked product/code changes: stop and ask before syncing or moving files.
   - Untracked non-harness files: stop and ask unless they are clearly generated by the current run.
   - Only generated `.harness/*` artifacts from an older run: quarantine them non-destructively.
3. **Validate `.harness` belongs to the current run before reuse.** Check any existing
   `.harness/project.json`, `.harness/plan.json`, `.harness/work-plan.md`, `.harness/dispatch.json`, and
   `.harness/units/*.json` for a matching source intent, source commit, tracking issue, scenario title, unit
   issue numbers, or current timestamp. If they point to a previous feature/run, they are stale even if
   syntactically valid. Unit status artifacts are run state too; old `WU-001.json` files can make a fresh run
   look already dispatched or completed.
4. **Quarantine stale harness state, never silently delete.** Move old generated artifacts to a timestamped
   archive such as `.harness/archive/<yyyyMMdd-HHmmss>/` or record that the human approved discarding them.
   Then recreate fresh `.harness` artifacts for the new intent.
5. **Output a readiness decision.**
   - `READY`: branch is current enough for the run, dirty state is understood, and `.harness` is either absent
     or verified as current.
   - `NOT-READY`: list the exact stale/divergent files and the safe next action.

## Honesty rules (hard)
- **Never infer current run state from file existence alone.** `.harness/plan.json` can be a stale artifact.
- **Never overwrite, reset, or delete human work to make the checkout clean.** Ask first.
- **Never continue planning or dispatch on a branch that is behind/diverged from the remote default branch**
  unless the human explicitly approves that base.
- **Never reuse a dispatch ledger or unit artifact from another run.** Stale ledgers and stale `.harness/units`
  files can double-dispatch, skip units, or report a previous scenario as complete.
